03

2025–2030 動態劇本：不是線性成長，而是三段式演進

Bernstein 圖表是 2025 與 2030 的靜態對比，但實際需求大概率不是線性增加，而是跟著 軟體 agent 平台成熟、硬體平台換代、企業 workflow 滲透 出現階段式跳升。

橫向捲動查看完整表格→
時間產業節奏CPU 需求變化
2025–2026Agentic AI 平台產品化CPU 需求先從雲端 agent runtime、企業試點、RAG、工具調用平台開始增加
2026–2027新一代 CPU / AI rack 平台量產Venice、Vera Rubin、GB300 / Rubin、Kyber 等平台可能讓 CPU:GPU ratio 明顯跳升
2028–2030企業 agent workflow 規模化Agent 進入 ERP、CRM、客服、資料分析、程式開發、資安、營運自動化，CPU compute layer 大規模部署

2025–2026 年的關鍵在軟體端。AWS Bedrock AgentCore 已 GA，AWS 將其定位為讓開發者把 agents 從 pilot 推向 full-scale production 的基礎平台；Microsoft Copilot Studio 在 2025 年宣布 autonomous agents 與 generative orchestration GA；Google Gemini Enterprise Agent Platform 則定位為企業建置、擴展、治理與最佳化 agents 的平台。 資料來源：AWS Bedrock AgentCore[^4]、Microsoft Copilot Studio[^5]、Google Cloud Gemini Enterprise Agent Platform[^6]

2026–2027 年的關鍵在硬體平台換代。AMD 已宣布 6th Gen EPYC「Venice」開始 production ramp，並稱 Venice 是第一個在 TSMC 2nm 進入 production ramp 的 HPC 產品；NVIDIA Vera CPU 則被官方定位為服務 agentic AI 與 reinforcement learning 的 CPU，負責 code execution、tool use、sandboxing、analytics、data pipelines 與 orchestration beyond the model。 資料來源：AMD Venice production ramp[^7]、NVIDIA Vera CPU[^8]

NVIDIA Vera Rubin 平台也被定位為 agentic AI 與 reasoning 時代的平台，目標是處理 multi-step problem-solving 與 massive long-context workflows；GB200 NVL72 則已經是 rack-scale liquid-cooled 設計，連接 36 顆 Grace CPU 與 72 顆 Blackwell GPU，形成 72-GPU NVLink domain。 資料來源：NVIDIA Rubin architecture[^9]、NVIDIA GB200 NVL72[^10]

所以 CPU:GPU ratio 從 0.25x 拉到 1.0x，較可能不是每年平均上升，而是跟著幾個節點跳升：

2025–2026：agent 平台 GA，企業試點增加
2026–2027：Venice、Vera Rubin、GB300 / Rubin rack、800VDC / Kyber 架構推進
2028–2030：企業 agent workflow 進入規模部署

NVIDIA 的 800 VDC 架構也暗示 AI data center 正從傳統 54V rack power 走向更高功率密度。官方指出 800 VDC 可降低轉換階段、減少銅用量與線纜體積，並支援更高 compute density。 資料來源：NVIDIA 800 VDC architecture[^11]

04

為什麼 Agentic AI 會更吃 CPU？

4.1 傳統 LLM：CPU 是 host，GPU 是主角

傳統 chatbot 型 AI 流程比較單純：

使用者輸入 prompt
→ tokenizer
→ GPU 跑模型推理
→ 產生回答

CPU 在這種架構中主要做：

橫向捲動查看完整表格→
CPU 工作說明
Host control啟動與管理 GPU 任務
Tokenization文字切 token
Data movement搬移資料
Scheduling任務排程
Networking與外部服務溝通

所以傳統 AI server 容易形成 GPU-heavy 架構。

4.2 Agentic AI：CPU 變成任務編排與資料處理中心

Agentic AI 不只是回答問題，而是會把任務拆成多步驟：

理解任務
→ 拆解子任務
→ 查資料
→ 呼叫工具
→ 讀資料庫
→ 執行程式
→ 驗證結果
→ 多輪修正
→ 輸出或執行動作

這讓 CPU 工作量大幅增加。

橫向捲動查看完整表格→
Agentic AI 工作為什麼會增加 CPU 負載
Tool calling呼叫 API、搜尋、資料庫、Python interpreter、內部工具
Orchestration多 agent、多步驟、多模型、多流程任務調度
RAG檢索、排序、向量資料庫、資料前處理
Enterprise workflow權限、交易、ERP / CRM / ITSM 連接
Memory / context management長上下文切分、快取、壓縮、狀態管理
Security / compliance權限驗證、資料隔離、審計記錄
Network I/O多節點、多服務、多資料來源通訊

一篇針對 agentic AI 的 CPU-centric 研究指出，agentic AI framework 在 LLM 上加入 decision-making orchestrator 與外部工具，使模型從被動文字回答變成能 plan、call tools、remember past steps 的系統；研究也觀察到工具處理在 CPU 上最高可占總延遲 90.6%，大 batch 下 CPU dynamic energy 最高可占總 dynamic energy 44%。 資料來源：arXiv CPU-centric agentic AI study[^12]

這就是為什麼 CPU 在 agentic AI 中不是單純「搭配 GPU 的 host」，而是變成 workflow runtime 的核心。

這裡最敏感的變數是 inference CPU:GPU ratio。AI capex 從 3.0 兆到 4.0 兆美元當然重要，但 CPU TAM 真正大幅變動的原因，是每單位 inference GPU 旁邊需要多少 CPU orchestration、data processing、memory、I/O 與 runtime 管理。

可以把 Bernstein 的三個情境理解成：

橫向捲動查看完整表格→

換句話說：

CPU TAM 的核心不是「AI capex 有多大」，而是「AI capex 裡有多少真的變成 agentic inference，以及每單位 inference GPU 需要多少 CPU」。

06

軟體層是 CPU 需求能否放大的關鍵變數

CPU 需求並不只由硬體世代決定，也會被 agent framework、inference serving、KV cache、RAG 與 enterprise workflow 軟體架構影響。

橫向捲動查看完整表格→
軟體層代表對 CPU 需求的影響
Agent orchestrationLangGraph、CrewAI、AutoGen、Copilot Studio、Bedrock AgentCore多步驟任務、工具調用、狀態管理、multi-agent 協作增加 CPU / memory / I/O 負載
Inference servingvLLM、TensorRT-LLM批次排程、KV cache、prefill / decode 分離、GPU utilization 決定 CPU 是否成為瓶頸
Memory / cache optimizationKV cache offloading、KV reuse、CPU / disk offload可能提高 CPU memory / I/O 角色，也可能降低重複運算
Enterprise data planeRAG、database connector、API、MCPCPU 從 host 變成 workflow / data plane 的一部分

vLLM Production Stack 官方文件指出，KV cache offloading 可將大型 KV cache 從 GPU memory 移到 CPU 或 disk，以提高 KV cache hit 的可能性。這代表 CPU / system memory 在長上下文與多輪 agent 任務中可能扮演更重要的 memory tier。 資料來源：vLLM KV cache offloading documentation[^13]

但這也形成一股反向力量：如果 GPU serving stack、DPU、BlueField、TensorRT-LLM、vLLM、KV cache reuse / offloading 把更多資料移動、storage、networking、cache 管理與 serving optimization 做得更有效率，general-purpose CPU 的邊際需求可能被部分壓低。NVIDIA Vera Rubin 平台本身就不是只靠 CPU，而是把 Vera CPU、Rubin GPU、NVLink、BlueField、Spectrum-X 等整合成完整異質運算架構。 資料來源：NVIDIA Rubin architecture[^9]

因此，未來 CPU 需求的形狀會分成兩塊：

會增加 CPU 的部分：
agent orchestration、tool call、RAG、API、資料處理、企業 workflow、sandbox execution

可能降低 CPU 邊際需求的部分：
DPU offload、GPU-native serving、KV cache optimization、specialized accelerator、network/storage offload

結論是：CPU 需求會增加，但不會平均分布在所有 CPU 任務上，而會集中在 agent runtime、workflow orchestration、memory / I/O、資料處理與企業應用整合。

07

全球 AI 伺服器 CPU 競爭格局

7.1 AMD：x86 server CPU 份額上升，EPYC 是最直接受惠者之一

AMD 的 server CPU 主力是 EPYC。6th Gen EPYC「Venice」已進入 production ramp，AMD 表示這是與 TSMC 2nm 技術合作的重要里程碑，也是業界第一個進入 TSMC 2nm production ramp 的 HPC 產品。 資料來源：AMD Venice production ramp[^7]

依 Mercury Research 數據公開轉述，AMD 在 2026 Q1 的 server x86 CPU unit share 達 33.2%，server x86 CPU revenue share 達 46.2%，顯示 AMD 不只出貨量提升，也在高階產品與 ASP 上取得更高收入占比。 資料來源：Tom's Hardware / Mercury Research 公開轉述[^14]

所以在 x86 內部，CPU TAM 擴大不是 AMD 與 Intel 平均受惠。AMD 的份額與收入占比正在上升，因此它對 agentic AI CPU 需求的邊際彈性更大。

Intel Xeon 仍是傳統 data center server CPU 的代表。Intel 官方將 Xeon 定位為支援 AI、HPC 與 data center workload 的處理器，強調 enterprise-class reliability 與 manageability。 資料來源：Intel Xeon 官方產品頁[^15]

Intel 的優勢是：

AMD EPYC 在 x86 server 裡提高份額
Arm / CSP 自研 CPU 正在吃掉部分 merchant CPU 空間

因此 Intel 仍會受惠於 AI server CPU 需求回升，但相較 AMD，其增長彈性更取決於 Xeon 平台更新速度、AI host CPU 定位，以及是否能守住企業與雲端基礎盤。

7.3 NVIDIA：從 GPU 公司變成 CPU + GPU + DPU + networking + rack-scale AI infrastructure 公司

NVIDIA 已經不是單純 GPU 公司。GB200 NVL72 官方資料顯示，它是一個 rack-scale liquid-cooled system，連接 36 顆 Grace CPU 與 72 顆 Blackwell GPU，形成 72-GPU NVLink domain。 資料來源：NVIDIA GB200 NVL72[^10]

NVIDIA Vera CPU 則更直接切入 Bernstein 圖表的 agentic AI CPU 主題。NVIDIA 官方表示，Vera CPU 是為 agentic AI 與 reinforcement learning 背後的 CPU 工作設計，包括 code execution、tool use、sandboxing、analytics、data pipelines 與 orchestration beyond the model。 資料來源：NVIDIA Vera CPU[^8]

agent runtime processor
workflow orchestration processor
